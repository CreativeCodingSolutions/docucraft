<p class="leading-relaxed text-muted-foreground"><strong>13% of open-source repos</strong> have a chronic PR description problem — 30% or more of their pull requests lack meaningful descriptions. That's 1 in 8 repositories.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground">And across all repos, 1 in 20 PR descriptions is literally empty. Zero characters. Just a title and a merge.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">The Methodology</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground">I scanned 175 open-source repositories (primarily GitHub Actions and CI tooling, stars ranging from 10 to 5,000+) using automated analysis of their recent PR history. For each repo, I examined the last 30 pull requests and classified descriptions by quality:</p>
<!-- -->
<li class="ml-6 list-disc text-muted-foreground"><strong>Weak:</strong> Body is empty, under 20 characters, or contains only placeholder text (&quot;fixes&quot;, &quot;bump&quot;, &quot;wip&quot;, &quot;update&quot;, etc.)</li>
<li class="ml-6 list-disc text-muted-foreground"><strong>Poor:</strong> Under 150 characters — technically present but functionally useless for review</li>
<li class="ml-6 list-disc text-muted-foreground"><strong>Good:</strong> Over 150 characters with meaningful context</li>
<!-- -->
<p class="leading-relaxed text-muted-foreground">Repos with ≥30% weak PRs across their last 30 PRs were flagged.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">The Worst Offenders</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground">Some repos had weak PR rates of <strong>87%, 63%, and even 100%</strong> — every single recent PR had an empty or near-empty description. Here's the full breakdown of the 23 flagged repos:</p>
<!-- -->

<p class="leading-relaxed text-muted-foreground">And 3 repos at <strong>100%</strong> — every single PR had zero description.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">What &quot;Weak&quot; PRs Look Like</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground">Here are real examples from popular repos (5,000+ stars):</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>Example 1 — Empty description, 6 files changed:</strong></p>
<p class="leading-relaxed text-muted-foreground">PR titled &quot;Update the latest release&quot; in softprops/action-gh-release.</p>
<p class="leading-relaxed text-muted-foreground">6 files modified (+90/-35 lines). Description: nothing.</p>

<p class="leading-relaxed text-muted-foreground"><strong>Example 2 — Single line that adds no context:</strong></p>
<p class="leading-relaxed text-muted-foreground">PR titled &quot;feat: update action to use node24&quot; in softprops/action-gh-release.</p>
<p class="leading-relaxed text-muted-foreground">3 files modified (+179/-160 lines). Description: &quot;Updates for vitest and esbuild&quot;</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>Example 3 — Empty description, README change:</strong></p>
<p class="leading-relaxed text-muted-foreground">PR titled &quot;mention Docker GitHub Builder in the README&quot; in docker/build-push-action.</p>
<p class="leading-relaxed text-muted-foreground">1 file modified. Description: nothing.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground">These aren't obscure projects. They're some of the most popular GitHub Actions repositories on the platform with 5,000+ stars each.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">Why This Is a Real Problem</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>1. Reviewers waste 15-30% more time</strong> on PRs with poor descriptions. Without context, a reviewer has to reconstruct the author's intent from the diff alone — which is like reading a book starting from page 50.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>2. New contributors face a steeper learning curve.</strong> Open source thrives on lowering barriers to entry. But when PRs lack context, new contributors can't learn from examples of how changes are structured and explained.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>3. Changelogs are harder to generate.</strong> Projects that rely on automated changelog generation produce lower-quality output when PR descriptions are sparse.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>4. The bus factor grows.</strong> Only the PR author understands the change's full context. Months later, even they may not remember why a particular change was made.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">Why Do Open Source PRs Lack Descriptions?</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground">Several patterns explain the gap:</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>The &quot;I'll add it later&quot; trap.</strong> Contributors push code first intending to write a description, then merge when CI passes and forget.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>Perceived time cost.</strong> Writing a good PR description takes 2-5 minutes. When you're contributing to 5 different projects, that time adds up. But the 30 minutes a reviewer saves more than justifies it.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>No enforced standard.</strong> Most repos don't have PR templates or automated checks for description quality. GitHub has a &quot;require PR description&quot; setting, but few repos enable it.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground"><strong>Assumed context.</strong> &quot;If you're on this team, you know what this change is about.&quot; This assumption breaks down as teams grow and rotate.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">What We Measured vs. What Good Looks Like</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground">The average PR description across our sample was <strong>~400 characters</strong> — roughly two short sentences.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground">Compare that to auto-generated descriptions from structured analysis of the same PRs: AI-generated descriptions consistently identify all changed files, categorize changes by type (feature, fix, refactor, dependency), and explain the rationale — something human contributors often skip.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">The Fix Is Already Here</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground">This isn't a hard problem to solve. The tools exist today:</p>
<!-- -->
<li class="ml-6 list-decimal text-muted-foreground"><strong>PR templates</strong> — GitHub supports them natively. Just add <code>.github/PULL_REQUEST_TEMPLATE.md</code>.</li>
<li class="ml-6 list-decimal text-muted-foreground"><strong>Automated enforcement</strong> — Add a CI check that rejects PRs with empty descriptions.</li>
<li class="ml-6 list-decimal text-muted-foreground"><strong>AI-assisted generation</strong> — Tools that analyze the diff and generate a structured description automatically.</li>
<!-- -->
<p class="leading-relaxed text-muted-foreground">The third option is where the biggest time savings come from, because it doesn't ask contributors to change their habits — it fills the gap automatically.</p>
<!-- -->
<h2 class="mt-10 mb-4 text-2xl font-bold tracking-tight">The Bottom Line</h2>
<!-- -->
<p class="leading-relaxed text-muted-foreground">13% of open-source repos have a chronic PR quality problem. 5% of all PRs have zero description. Reviewers spend 15-30% more time compensating.</p>
<!-- -->
<p class="leading-relaxed text-muted-foreground">This is a $10,000 problem with a $10 solution. The tooling exists. The only missing thing is the habit.</p>
